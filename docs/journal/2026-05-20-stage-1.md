## Why we did this

The original milesGPT was a 400-line `src/index.js` that:

1. Served a static HTML form
2. On submit, took the text, embedded it via Workers AI
3. Vector-searched a D1 notes table via Vectorize
4. Concatenated the top match + question into a system prompt
5. Called `llama-3-8b-instruct` once
6. Returned the full response as a server-rendered HTML page

It worked, but it was a Q&A box, not an assistant. It had no memory, no
streaming, no tool use, no way for the model to write back to its own
context, and no resilience to anything going wrong mid-call.

[Cloudflare's "Project Think"](https://blog.cloudflare.com/project-think/),
announced ~5 weeks ago, ships an opinionated `Think` base class on top of
their Agents SDK that wires up the entire agentic-loop infrastructure
(durable execution, persistent sessions, tool use, streaming, sandboxed
filesystems) as a Durable Object. Rebuilding milesGPT on top of it was a
massive reduction in code (`src/server.ts` is now ~85 lines including
comments) for a massive increase in capability.

## The five commits, in order

### 1. `76bf118` — Rebuild milesGPT on Project Think (the big one)

The entire scaffolding: TypeScript everywhere, Vite for the client, the
`MilesGPT` DO class, the React chat UI, the migration endpoint. Net diff:
`+5,242 / −1,971` lines (mostly `package-lock.json` churn; real code is
~600 lines).

The minimal agent that came out of this commit:

```typescript
// src/server.ts (excerpt)
export class MilesGPT extends Think<Env> {
  chatRecovery = true;

  getModel() {
    return createWorkersAI({ binding: this.env.AI })(
      "@cf/moonshotai/kimi-k2.6"
    );
  }

  configureSession(session: Session) {
    return session
      .withContext("soul", {
        provider: { get: async () => NIMBUS_SOUL }
      })
      .withContext("memory", {
        description: "Durable facts about Miles...",
        maxTokens: 2000
      })
      .withCachedPrompt();
  }
}
```

**What `Think` gives you for free** (this is the whole point):

- Per-DO SQLite, hibernating at zero idle cost
- Session-tree message storage (forking, search, compaction)
- Streaming chat over WebSocket via `useAgentChat`
- Durable execution — turns wrapped in a fiber, recoverable across crashes
- A Workspace filesystem at `this.workspace`, auto-exposed to the model as
  tools (`read`, `write`, `edit`, `find`, `grep`, `list`, `delete`)
- A `set_context` tool the model uses to update its own context blocks

**The "context block" concept** deserves its own note. Think models persistent
agent memory as named sections of the system prompt with two roles:

- **`soul`** — your personality / operating instructions. Static, provider-fed.
- **`memory`** — token-bounded, **the model writes to it itself** via the
  built-in `set_context` tool. This is how Nimbus remembers things across
  sessions without you wiring any external store.

That's the whole memory mechanism. Three lines of config in `configureSession`
and the agent now has persistent self-curated memory.

**The client** (`src/client/App.tsx`):

```typescript
const agent = useAgent({ agent: "MilesGPT", name: "miles" });
const { messages, sendMessage, status } = useAgentChat({ agent });
```

Two hooks. That's the entire integration. `useAgentChat` speaks the same
WebSocket protocol Think's base class implements, so messages stream
token-by-token with no custom transport code on either side.

**Routing** (`src/server.ts`):

```typescript
export default {
  async fetch(request, env) {
    return (
      (await routeAgentRequest(request, env)) ||
      new Response("Not found", { status: 404 })
    );
  }
} satisfies ExportedHandler<Env>;
```

`routeAgentRequest` handles the entire `/agents/<class>/<name>` URL space —
WebSocket upgrades, HTTP fallback, sub-agents, MCP callbacks. We add admin
routes before it; everything else falls through to the agent layer or the
static assets.

**Bindings** (`wrangler.jsonc`):

```jsonc
{
  "ai": { "binding": "AI" },
  "durable_objects": {
    "bindings": [{ "class_name": "MilesGPT", "name": "MilesGPT" }]
  },
  "migrations": [
    { "tag": "v1", "new_sqlite_classes": ["MilesGPT"] }
  ],
  "assets": {
    "not_found_handling": "single-page-application",
    "run_worker_first": ["/agents/*", "/admin/*"]
  }
}
```

The migration block is critical: any DO that uses SQLite storage must
declare itself in `new_sqlite_classes`. You can't deploy without it.

### 4. `67b0b4b` — Stage 1 testing framework (the second big one)

Three test layers, built before going any further:

**Layer 1 — pure-logic unit tests** (`test/unit/`, plain Vitest, ~100ms).
Tested the migration loop's contract in isolation:

```typescript
// test/unit/migrate-notes-logic.test.ts (now deleted post-clean-slate)
it("writes one file per row, using /notes/<id>.md as the path", async () => {
  const importNote = vi.fn().mockResolvedValue(undefined);
  const result = await migrate(
    [{ id: 1, text: "buy milk" }, { id: 2, text: "ring mum" }],
    importNote
  );
  expect(importNote).toHaveBeenNthCalledWith(1, "/notes/1.md", "buy milk");
  expect(result).toEqual({ migrated: [1, 2], count: 2 });
});
```

**Layer 2 — integration tests** (`test/integration/`, `@cloudflare/vitest-pool-workers`,
~10s total). Real `workerd`, real DO, real local SQLite. No Workers AI
calls — those would cost money and be flaky. The pool gave us:

```typescript
// test/integration/_helpers.ts
import { env } from "cloudflare:test";

export function nimbusStub() {
  return env.Nimbus.get(env.Nimbus.idFromName("miles"));
}
```

That `env` import is the magic — it's the real `Env` from `wrangler.jsonc`,
hot-wired by `vitest-pool-workers` into the test process. Tests look like:

```typescript
it("round-trips a simple file", async () => {
  const stub = nimbusStub();
  await stub.writeNote("/test/simple.md", "hello");
  expect(await stub.readNote("/test/simple.md")).toBe("hello");
});
```

That `stub.writeNote(...)` is a real DO RPC call, hitting a real Workspace
backed by a real local SQLite database, running inside a real `workerd`.
This level of fidelity in a local test loop is genuinely new — there was no
way to do this before `vitest-pool-workers`.

**Layer 3 — live HTTP/WebSocket smoke** (`scripts/smoke.mjs`, against
deployed URL). Two modes:

```bash
npm run smoke           # HTTP only, ~2s, $0 — checks the deploy is alive
npm run smoke -- --ws   # also opens a real WebSocket, sends "ping",
                        # waits for any streamed reply (proves model works)
```

The WS smoke sends a real `cf_agent_chat_messages` frame:

```javascript
ws.send(JSON.stringify({
  type: "cf_agent_chat_messages",
  messages: [{
    id: `smoke-${Date.now()}`,
    role: "user",
    parts: [{ type: "text", text: "ping" }]
  }]
}));
```

If any message comes back within 30s, we pass.

**Bug the tests caught immediately:** the very first run of the integration
tests revealed that `importNote()` was crashing with
`Cannot read properties of undefined (reading 'writeFile')`. The reason:
Think's `onStart` is async, and it's the thing that initialises
`this.workspace`. Custom RPC methods that run before `onStart` completes
see undefined fields and throw. The fix:

```typescript
async importNote(path: string, content: string): Promise<void> {
  await this.__unsafe_ensureInitialized();  // ← gate every custom RPC
  await this.workspace.writeFile(path, content);
}
```

**This bug would have stayed dormant in production** until someone happened
to hit `/admin/migrate-notes` before any chat message had warmed up the DO
— at which point the migration would silently fail with a 500. The test
framework paid for itself on the first run.

**Naming pitfall, found by the WS smoke test:** the `agents` library converts
class names to URL segments via `camelCaseToKebabCase`, character-by-character.
So `MilesGPT` becomes `miles-g-p-t`, not the more obvious `miles-gpt`. The
WS test failed until I matched that:

```javascript
const wsUrl = TEST_URL.replace(/^http/, "ws") + "/agents/miles-g-p-t/miles";
```

This single observation was the genesis of the next commit — renaming the
class to a single word eliminates the problem.

### 5. `2815089` — Clean slate: rename DO MilesGPT → Nimbus, drop migration code

With Stage 1 validated, retire the scaffolding that existed only to bring
data over from the old D1 + Vectorize milesGPT:

- Rename `MilesGPT` → `Nimbus` (cleaner, dodges the kebab routing pitfall)
- Add a v2 migration to wipe the old DO's SQLite + create the renamed class
- Drop `D1` + `Vectorize` bindings entirely
- Remove `/admin/migrate-notes` + `importNote` + `migrateNotes()`
- Delete the migration-specific tests
- Update the smoke script's WS path from `/agents/miles-g-p-t/miles` to
  `/agents/nimbus/miles`

**The most interesting bit here is the DO class rename mechanic.** You
can't just rename a DO in Cloudflare — the existing storage stays bound to
the old class name. The way to do a "rename + fresh start" is via the
migration chain:

```jsonc
"migrations": [
  // v1 created the now-deleted MilesGPT class. Kept in history because
  // Cloudflare requires the full migration chain to deploy.
  { "tag": "v1", "new_sqlite_classes": ["MilesGPT"] },

  // v2 — clean slate: delete the old DO class (wiping its SQLite),
  // create the renamed Nimbus class fresh.
  {
    "tag": "v2",
    "deleted_classes": ["MilesGPT"],
    "new_sqlite_classes": ["Nimbus"]
  }
]
```

Cloudflare runs every migration tag your account hasn't yet seen, in order,
on `wrangler deploy`. Once v2 ran in production, the old `MilesGPT`
storage was gone and the new `Nimbus` started pristine. No "drop database"
button — the migration chain *is* the mechanism.

**Net diff:** `+122 / −419` lines. The clean slate was bigger than the
clean-up — most of the deletion was the `migrate-notes` test suite (193
lines) and the rest was the migration code path itself.

## Concepts I learned today

### 1. **Agents-as-Durable-Objects is a fundamentally different scaling model**

In the old world, scaling a chat app meant: container per N concurrent
users, expensive when idle, plumbing for state externally.

In the new world, you spawn **one Durable Object per user (or per
conversation, or per task)**. The DO hibernates when idle — costs $0 —
and wakes on message. The marginal cost of "a Nimbus per person" is
effectively zero. Your `name` argument (`"miles"` in my case) is the only
addressing primitive.

### 2. **Code is just config — the agent's *capabilities* live in the bindings**

The `Nimbus` class is ~50 lines. It does almost nothing. The actual
capabilities — streaming, persistence, tool calling, durable execution,
memory, search, compaction — all come from extending `Think` and from the
declared bindings. The class is config, not logic.

### 3. **Context blocks are a real abstraction worth internalising**

`withContext("memory", { ... })` doesn't just put a string in the prompt.
It registers a *named section* of the system prompt that:

- The model can read (it sees it on every turn)
- The model can write (via the auto-injected `set_context` tool)
- Is bounded (`maxTokens: 2000`)
- Persists across hibernation

This is "long-term memory" implemented as durable structured prompt sections,
not as RAG over an embedding store. Much simpler, much cheaper, and good
enough for personal-assistant use cases.

### 4. **The Workspace is "filesystem-as-tool-belt"**

The old milesGPT had to manually orchestrate: text → embedding → vector
query → D1 lookup → prompt construction. Five steps of glue code.

The new Nimbus has zero of those. It has a Workspace (durable filesystem),
and Think auto-exposes `read`/`write`/`edit`/`find`/`grep`/`list`/`delete`
to the model as tools. When you ask Nimbus a question, the *model decides*
whether to grep its files, find a path, read content, and respond. No
explicit RAG plumbing. For a few hundred notes, grep beats vector search
on cost, latency, and clarity.

### 5. **`vitest-pool-workers` is genuinely new infrastructure**

You can now write integration tests that run *inside a real `workerd`*,
with real Durable Objects, real SQLite, real bindings — locally, fast,
deterministic, free. The bridge is `import { env } from "cloudflare:test"`
inside test files. This collapses a category of bug I used to only find in
production.

### 6. **Async `onStart` is a trap if you don't gate your custom RPCs**

Think's `onStart` runs `await Workspace.create(...)` and a bunch of other
init. Until that resolves, `this.workspace` is undefined. Custom RPC
methods are called by the runtime *immediately* on the DO instance,
without waiting for `onStart`. The fix:

```typescript
await this.__unsafe_ensureInitialized();
```

at the top of every custom RPC that touches Think state. The
"unsafe" prefix is just because the agents lib doesn't want you reaching
for it casually — using it here is correct and necessary.

### 7. **DO renames are migrations, not edits**

You can't rename a Durable Object class in place. The class is the *key*
its SQLite storage is bound to. To rename (or wipe), you ship a migration
that simultaneously:

```jsonc
{ "deleted_classes": ["OldName"], "new_sqlite_classes": ["NewName"] }
```

Old storage is gone. New storage is pristine. The migration chain in
`wrangler.jsonc` is permanent history — you can't remove old tags even
once their classes are deleted.

### 8. **Class names round-trip through kebab-case for routing**

`agents` exposes DOs at `/agents/<class-kebab>/<name>` via
`camelCaseToKebabCase`. It splits on every capital letter, so `MilesGPT`
becomes `miles-g-p-t`. Single-word class names (`Nimbus`) make the URL
predictable. Worth remembering when picking a class name.

### 9. **SPA `not_found_handling` makes routing tests subtle**

With `"not_found_handling": "single-page-application"`, any path the
Worker doesn't handle returns `index.html` + 200 OK so the client-side
router can take over. This is correct behaviour — but it means a smoke
test for "does this unknown route 404?" will *always* fail. The right
assertion is "does the SPA fallback actually serve the React shell?":

```javascript
const res = await fetch(`${TEST_URL}/some-random-path`);
assert(res.status === 200);
const html = await res.text();
assert(html.includes('id="root"'));  // ← it's the React shell
```

### 10. **The `useAgentChat` protocol is portable**

It's just `cf_agent_chat_*` JSON frames over a WebSocket. The smoke test
in `scripts/smoke.mjs` proves you can speak it from raw Node without
React, the library, or any helpers. Useful to know for future automation
(scheduled jobs, webhooks, integrations).

## What's deliberately not done yet (the Stage 2+ backlog)

| Stage | Capability | Why we'd want it |
| --- | --- | --- |
| 2 | R2 spillover for the Workspace; semantic search as a tool | Once the workspace grows past a few hundred files, grep alone gets slow / lossy. |
| 3 | **Codemode** — the model writes JS that runs in a Dynamic Worker instead of chaining tool calls | Biggest capability jump available. 100x fewer round-trips on multi-step tasks. |
| 4 | Browser Run + Cloudflare Sandbox tiers; sub-agents | Web automation; isolated parallel work. |
| 5 | Self-authored extensions (model writes its own tools at runtime) + GitHub auth | The thing that makes Nimbus genuinely improve over time. Auth becomes mandatory once the URL is shared. |

Stage 3 (codemode) is the next obvious move. The infrastructure for it
already exists (`@cloudflare/codemode`, `@cloudflare/worker-bundler`) and
the pattern is well-documented in the `examples/assistant` repo.

## Risks and known issues carried forward

1. **`@cloudflare/think@0.7.x` is preview.** The docs say "API surface
   stable but may evolve." A future minor could break us. Mitigation: pin
   exact versions (`^0.7.1`), watch the changelog.

2. **No authentication.** Anyone with the deployed URL can chat with
   Nimbus and see anything in its workspace. Fine on a `*.workers.dev`
   URL nobody's been given. Not fine on a real domain. The `examples/assistant`
   pattern using GitHub OAuth is the obvious template.

3. **`close timed out after 2000ms` warning on `npm test`.** Cosmetic.
   Tests pass, exit code 0, but Vitest hangs briefly on teardown due to
   open MCP-related connections in `workerd`. Not blocking. Track if a
   future `vitest-pool-workers` release fixes it.

4. **One DO instance for everyone.** `name: "miles"` means everyone hits
   the same Nimbus. That's *fine* for me-only use, but it's an artifact —
   once auth lands, the DO name should become the user identity.

5. **The smoke script defaults to my live URL.** If you fork this, set
   `DEFAULT_URL` in `scripts/smoke.mjs` to your own, or rely on
   `TEST_URL=...`. Not a bug, but easy to miss.
